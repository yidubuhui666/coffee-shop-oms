# 优化建议（项目层面，非代码）

日期：2026-05-29
背景：课程作业，核心卖点是「多表查询关系」，需答辩演示。

## 1. 文档与可读性（性价比最高）
- **README 重写**：项目简介 / 功能列表 / 技术栈 / 运行步骤 / 账号 + 功能截图。
- **数据库 ER 图**：画出订单、商品、原料、会员折扣等表的外键与多对多关系，
  放进 README/docs，答辩直观加分。
- **课程报告素材**：整理已解决的问题（密码泄露处理、N+1 优化、Blueprint 重构、
  分页/错误页/测试），作为报告「问题与解决」章节。

## 2. 答辩 / 演示准备
- 设计一条演示主线：登录 → 下单（库存自动扣减）→ 改状态加积分 → 看多表 JOIN 报表。
- 确认统计页数据充足（当前 150 单足够）。

## 3. 协作流程（许跃骞 + 杨帆）
- 改用「分支 + Pull Request」，合并前互相 review，避免覆盖。
- 保持中文清晰 commit 规范。

## 4. 功能完整性（加分项）
- 报表导出 Excel/CSV。
- 操作日志表（谁、何时、改了什么）——又一组表关系，契合主题。
- 角色权限在 UI 上更明显（不同角色不同按钮）。

## 5. 工程健壮性（可选）
- 给 orders.order_time、order_items.product_id 等常查字段加索引。
- 密码哈希存储（Werkzeug 已在依赖里）。

## 推荐优先级
先做 1（文档 + ER 图）和 2（演示准备），观感提升最大。
